1. What is Java SE?

Java SE (Standard Edition) masaüstü kompüterlərdə və serverlərdə Java proqramlarını inkişaf etdirmək və yerləşdirmək üçün istifadə olunan Java API (Tətbiq Proqramlaşdırma İnterfeysləri) toplusudur. Java SE Java proqramlarının işlənib hazırlanması və icrası üçün hərtərəfli kitabxanalar, alətlər və iş vaxtı mühitləri təqdim edir. Java SE-yə Java proqramlarının işlənib hazırlanması və sazlanması üçün alətlər toplusunu təmin edən Java İnkişaf Dəsti (JDK) və Java proqramlarını idarə etmək üçün istifadə olunan Java Runtime Environment (JRE) daxildir. Java SE masaüstü proqramlar, mobil proqramlar və veb proqramlar hazırlamaq üçün geniş istifadə olunur. O, maliyyə, səhiyyə, təhsil və s. daxil olmaqla, geniş sənaye sahələrində möhkəm və miqyaslana bilən tətbiqlər yaratmaq üçün güclü və çevik platforma təqdim edir.

2. What is the difference between the JDK and the JRE and JVM?

JDK və JRE arasındakı fərq nəzərə alındığında, JDK (Java Development Kit) bir proqramçıya Java proqramları hazırlamaq və sınamaq üçün lazım olan bütün alətləri təqdim edir. Bu alətlər arasında kompilyator, debager və digərləri daxildir.
JRE (Java Runtime Environment) isə bir Java proqramının işlədilməsi üçün lazım olan minimum proqramlar və sənədlər birləşməsidir. JRE bir Java proqramını çalışdırmaq üçün lazım olan bütün tələbləri ötürür, amma proqramın hazırlanması üçün alətlər təqdim etməz.
JDK, proqramların hazırlanması və sınanması üçün tələblər olan bütün alətləri təqdim edir, o cümlədən JRE-dən də ibarətdir. JRE isə yalnız Java proqramlarının işlədilməsi üçün lazım olan minimum sənədləri təmin edir.
"Sənədlər" burada Java proqramının işlədilməsi üçün lazım olan faylları nəzərə alır. Bu fayllar arasında Java Virtual Machine (JVM) üçün tələb olunan Java class faylları, Java runtime kitabxanası, Java API dokumentasiyası və s. kimi fayllar daxildir. Bu fayllar Java proqramının düzgün işləməsi üçün lazım olan əsas fayllardır və JRE-nin tərkibində olan fayllardır
JVM (Java Virtual Machine) : Javanın ən cəzbedici xüsusiyyəti bütün əməliyyat sistemlərində istifadə edə bilməyimizdir. Bir dəfə kodla, hər yerdə işlət. Bunu bacaran isə JVM-dir. Yazdığımız “.java” sonluqlu kodları kompilasiya edib “.class” sonluqlu “byte code” fayllar yaradır. Sadəcə o “byte code”-lar əməliyyat sisteminin başa düşəcəyi dildə deyil. Burda işin içinə bizim balaca JVM girir. Həmin kodu komputerin anlaya biləcəyi dilə çevirir. Burda bir məsələ var ki, hər bir əməliyyat sistemlərinin özünə uyğun JVM-i var ki, o da həmin əməliyyat sisteminə uyğun “byte code” yazır. JVM üçün biz JRE yükləməliyik. Hardan çıxdı indi JRE (Java Runtime Environment)?
JRE = JVM + digər lazimi kitabxanalar.
JRE yüklüyüb kod yazmağa keçə bilərik? Xeyr! Çünki JRE içində kod yazıb kompilasiya etməyimiz üçün lazım olan “compiler”, “debugger” və s. yoxdur. Bunun üçün JDK yükləməliyik.
JDK (Java Development Kit) = JRE + “Java Compiler”, “Debug Tool” və başqa lazımlı alətlər.
Qısaca, gözəl insan, sənə Java kodu yazıb işlədə bilmək üçün JDK lazımdı JDK!.
JVM nədir? İş prinsipi.
JVM(Java Virtual Machine)- adında da məlum olduğu kimi virtual maşındır. JVM C++ ilə hazırlanmışdır. Hansı ki bu maşın əməliyyat sistemi ilə Java arasında olan bir mühitdir. Platformaya (Windows,Linux, Mac və s.) bağlı olaraq işləyir. Hər platformaya aid öz implementasiyası var. Bizim yazdığımız .java uzantlı faylları oxuya bilməz. Compile zamanı .class fayllar yaranır ki bu faylları oxuya və CPU-ya əmr göndərir. Belə ki .class fayllar bytecode-lara çevrilmiş fayllardır. JVM bu faylları oxuya bilir. Bu yazdıqlarım daha anlaşıqlı olsun deyə sadə bir sxem üzərində baxaq.
.java faylları JDK vasitəsilə daha doğrusu JDK içərsində yer alan compiler ilə compile edərək bytecode-lara çevirir. Daha Sonra isə JVM işə düşür və .class faylları oxuyur. Bu yanaşmaya görədə Java-da proqramı bir dəfə yazdıqdan sonra bütün platformalarda rahatlıqla işlətmək mümkündür.
JDK özündə JVM,JRE və Compiler-i saxlayır. Buna görə də əgər siz java-da proqram yazmaq istəyirsinizə mütləq şəkildə o sizin kod yazdığınız mühitə install olmalıdır. Əgər siz compile olunmuş kodu run etmək istəyirsinzsə JRE kifayət edir. Çünki JRE özündə JVM və javanın kitabxanalarını saxlayır.

Java-da sinif obyektlərin yaradılması üçün plan və ya şablondur, obyekt isə sinifin nümunəsidir. Sinif obyektin xassələrini və davranışlarını müəyyən edir.
O, məlumat üzvlərini (sahələrini) və sinifdən yaradılmış obyektlərin davranışını təyin edən metodları ehtiva edir. Bir sinif məlumat üzvləri üçün hər birinin öz dəyər dəsti olan bir neçə obyekt yaratmaq üçün istifadə edilə bilər. Digər tərəfdən, obyekt bir sinifdən yaradılmış konkret bir varlıqdır. Onun bir halı, davranışı və şəxsiyyəti var. Bir obyektin vəziyyəti məlumat üzvlərinin dəyərləri ilə, davranışı isə metodları ilə müəyyən edilir. Obyekt onu eyni sinifin digər obyektlərindən fərqləndirən unikal şəxsiyyətə malikdir. Daha sadə dillə desək, sinif obyektin necə görünəcəyini və necə davranacağını müəyyən edən plana bənzəyir, obyekt isə həmin planın həqiqi fiziki təzahürüdür. Məsələn, Car adlı bir sinfi nəzərdən keçirək. Bu sinifdə marka, model, il və rəng kimi məlumat üzvləri, həmçinin start, sürətləndirmə və dayandırma kimi üsullar ola bilər. Car sinifinin obyekti bu məlumat üzvlərinin hər biri üçün özünəməxsus qiymətlərə malik olacaq və metodlarla müəyyən edilmiş davranışları yerinə yetirə biləcək. Xülasə, sinif obyektlər yaratmaq üçün şablon və ya plandır, obyekt isə özünəməxsus dəyər və davranışlar dəsti olan bir sinif nümunəsidir.

Artıq biz, sinif,obyekt, metod,parametr anlayışlarını əvvəlki mövzulardan öyrənmişik. OOP-ni nə olduğunu bilirik. OOP Obyektyönümlü proramlaşdırma deməkdir. OOP dillərdə kod yazmaq üçün bir çox prinsiplərə əməl etmək lazımdır. OOP-nin əsas 4 təməl prinsipi var. OOP-nin prinsipləri aşağıdakılardır.

Abstraction(Mücərrədlik)- Sinif yaradarkən və bu sinfi digər bir sinifdən extend edən zaman bu sinif həmin sinfin də xassələrini özünə götürüdü. Beləliklə biz bir mücərrəd sinif yaratsaq o zaman bu sinfin də xüsusiyyətlərini child sinif özündə birləşdirəcək. Mücərrəd(abstract) sinifdə biz dəyişən və abstrakt metodlar təyin edirik. Yəni bir növ metodların ancaq giriş və çıxış şəklini təyin edirik. Əsas sinfimizi bu sinifdən extend edilməsi isə abstraksiya adlanır. Yəni əslində olmayan bir sinif yardıb burda metod və dəyişənləri təyin etmiş oluruq. Bir şeyi də nəzərimizdə saxlayaq ki abstract sinfin heç bir xüsusiyyəti private ola bilməz. Abstarct sinfidə heç bir iş görülmür sadəcə olaraq bir növ qanunlar qoyulur və bu qanunları varis yerinə yetirir. Eyni zamanda əgər abstract sinifdə dəyikşən təyin olunubsa o zaman varis sinifdə həmin dəyişən birbaşa istifadə olunur. Nümunəyə baxaq.
Abstrakiyanın bəzi üstünlükləri aşağıdakılardır:
Kodda qarışılığı aradan qaldırılmasına kömək edir.
2. Kodda dublikatlığı azaltır və təkar istifadə oluna bilməsini artırır.
3. Ancaq lazım olan xassələri göstərməklə application təhlükəsizliyini artırır.
Encapsulation-  Yadımızdadırsa biz access modifier mövzusunda bir cümlə işlətmişdik. Sinfin private parametrlerinə birbaşa müraciət mümkün deyil. Amma dolayı yolla onlara mürciət edə bilərik. Bu yolun adı kapsullamadır. Encapsulation təkcə sinfin paremtrlərinə access yox həmdə qəbul edilən və ya geri döndərilən informasiyanı da strukturlaşdırmağa kömək edir.

Inheritance - bunu bir DNA kimi fikirləşin. Atadan oğula ötürülən özəllik kimi. Bu Siniflərdə də eynidir. Necə ki insanda valdeyn-övlad əlaqəsi var siniflərdə də bu inheritance adlanır. Yəni bir sinif digərini özünə varis qəbul edir. Bir nümunə üzərindən baxaq. Tutaq ki mənim A sinfim və bir də B sinfim var.
Bir sinfi digər sinfə varis etmək extends  sözündən istifadə olunur. Javada bir sinif çoxlu sinfin üst(parent) sinfi ola bilər amma bir sinifin yalnız bir parent sinfi ola bilər. Yəni A extends B,C,D və olmaz.
Polymorphism(Polimorfizm, Çox formalılıq)- Abstarksiyaya oxşasa da abstraksiya deyil. Aralarında kəskin fərq var.
Abstract sinifdən obyekt yaradıla bilməz.
Abstract sinfin metodlarının içərisi boş olur. Sadəcə metodun skleti olur.
Amma polimorifzm-də əsas götürülən(parent) sinif real sinifdir. Abstract sinif deyil yəni. Polimorifzmdə parent sinfin bütün xassələri istifadə olunmaya bilər. Polimorfizmin əsas məqsədi tək obyekt üzərindən child və parent sinfin xüsusiyyətlərindən istifadə edə bilməkdir. Eyni zamanda lazım olan metodlari override edərək də istifadə edə bilərik. Polimorfizmə aid bir nümunəyə baxaq.
public class Animal { public int age; public void showAge() { System.out.println(this.age); } }class Dog extends Animal { public Dog() { this.age = 6; } @Override public void showAge() { System.out.println(this.age); } public void run() { System.out.println("Dog running"); } } public class Cow extends Animal { public Cow() { this.age = 10; } }public class Main { public static void main(String[] args) { Dog dog=new Dog(); dog.showAge(); dog.run(); Cow cow=new Cow(); cow.showAge(); } }
Burda biz Animal sinfi yaradıq və bu sindən əlavə 2 sinif extend etdik. showAge() metodunu Dog sinfində override edərək işlətdik. Amma digər sinifdə isə olduğu kimi saxladıq.

Access Modifiers in Java

Bir öncəki mövularda siniflərin,metodların,parametrlərin qarşısında public,private və s kimi sözləri gördük. Bəs bunlar nədir? Nə üçün istifadə olunur? Access modifiers javada icazə rolunu oynayır. Belə ki biz siniflərin medotların qarşısında yazmaqla o sinfi,metodu hardan çağıra biləcəyimizin icazəsini vermiş oluruq. Javada 4 növ access modifiers var.
Default
Private
Protected
Public
Default modifier-in keyword-ü yoxdur. Yəni əgər biz class A{ } demişiksə bu o deməkdir ki, defaultdur. Default sinifər yalnız eyni package daxilində bir birlərini görürlər.
Private - modifier siniflərdə olmur. bu modifier həmin sinfin parammetr və medtoların üçün keçərlidir. Əgər biz bir dəyişəni və ya metodu private olaraq təyin etmişiksə o yalnız həmin sinif daxilində işlənə bilər. Düzdür sinifdə təyin olunan private dəyişənə digər bir yolla digər siniflərdən dəyər almaq və ya ötürmək olur. Bu barədə OOP-nin prinsipləri mövzsunda danışacağıq.
Public -Adından da məlum olduğu kimi əgər biz public olaraq təyin etmişiksə onda bu dəyişən və ya sinfə proqramın istənilən yerindən(package) müarcət edə bilərik.

10. What is Stack and Heap ?

Heap-Stack bellekte(RAM)’ de bulunan iki ayrı bölgedir. Primitive tipler (int,float,double,short ,byte,decimal,long) yani value type(değer tip) stack’te tutulur. Bu değerler çalışmadan önce yani runtime öncesinde bilinmesi ve işletim sistemi tarafından yerlerinin ayrılması gereken değerlerdir. Bu yer ayırma kullanıcı tarafından aşılırsa stackoverflow hatası alınmaktadır.
Referans type(referans tip) referansları stack’te tutulurken değerleri heap’te tutulmaktadır. Java’da Wrapper tipler dediğimiz Integer, Long, Double, Boolean, Object, Person (kendi özel tipimiz) tiplerine karşılık gelmektedir.
Stack ve Heap Özellikleri
Stack LIFO mantığında çalışır. Yani son gelen ilk olarak çıkar.
Stack’de değer tipleri, pointer ve adresler saklanırken Heap’de ise referans değerleri saklanır.
Stack daha hızlıdır. Ulaşılmak istenen veriler ard arda sıralanmış olur.
Heap ortak olarak kullanılır ve uygulama başlatıldığında başlar.
Stack ve Heap bellekte (ram) bulunan mantıksal yapılardır. Primitif tip dediğimiz int, short, byte, long, decimal, double, float gibi tipler value type (değer tipi) olarak adlandırılır ve stack de tutulur. Çalışma zamanından (runtime) önce bu değerlerin bilinmesi gerekir ve işletim sistemi program çalışmadan önce stack de belirli bir yer ayırır eğer bu bölüm kodu yazan kişi tarafından aşılırsa stack taşma hatası ile (stack overflow) karşılaşılabilir.. Stack de veriler üst üste (LIFO) mantığında dizilir ve sırası gelmeden aradaki bir değer ile işlem yapılamaz. Class type (Sınıf tipi) değişkenler referans tiplerdir referans ettikleri model (referans) stack de değerleri ise heap de saklanır.
Heap ve Stack arasında ki en önemli farklardan birisi heap de veriler karışık bir şekilde saklanır stack de ise artan ya da azalan adres mantığında (big and little endian) çalışır. Buna bağlı olarak heap de ki bir veriye erişmek stack de ki bir veriye erişmeye göre daha maliyetli bir işlemdir. Başka bir farkı ise stack de ki veri hemen silinirken heap de ki veri Garbage Collector (Çöp Toplayıcı) algoritmasına bağlıdır.

Heap Və Stack Yaddaş Anlayışları.

Hər hansı bir proqram təminatının hazırlanması zamanı mühim məsələlərdən biri də kompüterin yaddaşından doğru istifadə olunmasıdır. Bizim yazdığımız kodlar işə düşərəkən kompüterin RAM-ına (Random Acces Memory) yüklənir və proqram icra olunarkən RAM-in nə dərəcədə istifadə olunması proqramın məhsuldarlığına təsir edir. Kompüterin RAM yaddaşı iki əsas hissəyə ayrılır: Heap və Stack yaddaşları.
Heap və Stack anlayışları aparat təminatı ilə deyil proqram təminatı ilə bağlı anlayışlardır. Bu məsələni göz ardı etmək olmaz. Yəni bu iki yaddaş sahəsi RAM-da hər hansı fiziki yaddaş sahəsi deyil. Sadəcə olaraq yazdığımız proqram icra zamanı dəyişənləri saxlamaq üçün istifadə etdiyi yaddaşı iki hissəyə ayırır. Heap və Stack yaddaşılarının təyin olunması birbaşa bizim proqramın öhtəliyində, yəni dolayısı ilə bizim öhtəliyimizdə olan bir şeydir.
Stack yaddaşı kod yazarkən dəyəri statik olaraq təyin olunan primitiv tipli dəyişənləri və obyekt referanslarını (referance) özündə saxlayır. Primitiv tipli dəyişənlərin istifadə edəcəkləri yaddaş sahəsi əvvəlcədən təyin olunur və yaddaşda müəyyən bir ünvana sahib olur. Belə dəyişənlərə əsasən lokal dəyişənlər aid edilir. Stack yaddaşında həmçinin primitiv tipli dəyişənlərdən istifadə edən funksiyalarda saxlanır. Stack yaddaşında saxlanan dəyişənlərə proqramın birbaşa müraciəti vardır və buna görə də stack yaddaşdan oxuma və yazma daha sürətlidir. Stack yaddaşın mənfi cəhəti onun həcminin limitli olmasıdır (ƏS`dən asılı olaraq).
Heap yaddaş sahəsi kodda elan olunan və proqramın icrası zamanı hər hansı bir mərhələsində dinamik olaraq dəyişə bilən və yaddaşda primitiv tiplərə nəzərən daha çox yer tutan obyektləri saxlamaq üçün ayrılır. Biz kodda hər hansı dəyişən elan edərkən onun dəyərini təyin etmədiyimiz zaman bir “pointer”, yəni bir referans yaratmış oluruq. Məsələni daha da açıqlamış olsaq, bir ünvan yaradırıq. Lakin yaratdığımız bu ünvan heç bir yeri göstərmir. Bu zaman yaratdığımız ünvan stack yaddaşında qeyd olunur. Proqramın icrası zamanı irəliləyən mərhələlərdə bu ünvana hər hansı bir dəyər təyin etdikdə bu dəyər artıq heap yaddaşında saxlanır və stack yaddaşında olan ünvan ilə əlaqələndirilir. Yəni biz həmin dəyəri çağırarkən stack yaddaşında yerləşən ünvana müraciət edirik və o da bizə heap`dəki dəyəri gətirir. Beləliklə, heap yaddaşa bizim birbaşa müraciət etməyimiz mümkün olmur. Buna görə də heap yaddaşa müraciət stack yaddaşa nisbətən daha ləngdir və heap yaddaşın həcmi stack yaddaşa nisbətən daha çox olur.
Məsələn, primitiv tiplərdən int (integer) tipə malik “a = 32” dəyişəni birbaşa 32 dəyəri ilə stack yaddaşda saxlanır. Digər tərəfdən referans tipə malik array (məs: b = array.array(‘i’, [1, 2, 3])) dəyişəninin qaşılığında bir array obyekti yarandığı üçün onu işarə edən referans stack yaddaşında, onun dəyəri olan obyekt isə heap yaddaşda saxlanır.
Heap və stack yaddaşlarının digər fəqli cəhəti isə odur ki, stack yaddaşı siz idarə etmirsiniz. Yaddaş CPU  tərəfindən səmərəli şəkildə idarə olunur. Yəni dəyişən proqram daxilində istifadə olunandan sonra əgər bir daha bizə lazım olmayacaqsa stack yaddaşından silinir. Artıq bizim bunun üçün əlavə nəsə etməyə ehtiyacımız qalmır. Amma heap yaddaş belə idarə olunmur. Lazımsız obyekt və dəyişənləri biz özümüz yaddaşdan təmizləməliyik. Əgər biz bunu etməsək proqramımız gərəksiz yerə yaddaşdan daha çox istifadə edəcək və məhsuldarlıq aşağı düşəcək. Bu problemi aradan qaldırmaq üçün bəzi proqramlaşdırma dillərində “Garbage Collection ” anlayışından istifadə edilir. Garbage collection heap yaddaşda referansdan qopmuş, yəni müraciət üçün heç bir ünvana sahib olmayan obyekt və dəyişənləri yaddaşdan silir. Heap`də yerləşən obyektlərə biz eyni zamanda bir neçə ünvan (referance) da bağlaya bilərik.

Java proqramçılarına obyektlər üçün yaddaşı əl ilə ayırmağa ehtiyac yoxdur, çünki Çöp Kollektoru bunu avtomatik idarə edir. Bununla belə, proqram tərəfindən ayrılan və istifadə olunan yaddaşın həcmini minimuma endirən səmərəli kodun yazılması vacibdir, çünki həddindən artıq yaddaş istifadəsi proqramın işinə və Zibil Toplayıcının səmərəliliyinə təsir göstərə bilər.

23. What is difference Abstract Classes vs. Interfaces ?

Bir sinif yalnız bir mücərrəd sinfi genişləndirə bilər, lakin birdən çox interfeys həyata keçirə bilər.
Abstrakt siniflərdə dəyişənləri ola bilər, lakin interfeyslər ola bilməz.
Abstrakt siniflərdə həm abstrakt, həm də qeyri-mücərrəd üsullar ola bilər, interfeyslərdə isə yalnız abstrakt metodlar ola bilər.
Abstrakt siniflərin konstruktorları ola bilər, lakin interfeyslər ola bilməz.
Mücərrəd siniflər alt siniflərin miras ala və genişləndirə biləcəyi əsas tətbiqi müəyyən etmək üçün istifadə edilə bilər, interfeyslər isə siniflərin həyata keçirməli olduğu metodlar toplusunu müəyyən etmək üçün istifadə olunur.
Mücərrəd sinif və ya interfeysdən istifadə etmək qərarına gəldikdə, həll etməyə çalışdığınız problemin xarakterini nəzərə alın. Əgər alt siniflərin miras ala və genişləndirə biləcəyi əsas tətbiqi müəyyən etmək lazımdırsa, abstrakt sinifdən istifadə edin. Əgər siz siniflərin həyata keçirməli olduğu metodlar toplusunu təyin etməlisinizsə, interfeysdən istifadə edin. Bəzi hallarda hər ikisindən istifadə etmək məqsədəuyğun ola bilər.
Interface'ler ve abstract class'lar, Java programlamasında farklı soyutlama mekanizmalarıdır ve farklı kullanım senaryolarına sahiptir. İşte Interface'ler ve abstract class'lar arasındaki temel farklar:
Kullanım Amacı: Interface'ler, bir sınıfın belirli bir davranışı uygulamasını sağlamak için kullanılır. Bir interface, sadece imzaları (signature) tanımlar ve uygulayıcı sınıfın bu yöntemleri gerçekleştirmesini gerektirir. Abstract class'lar ise genellikle ortak davranışı paylaşan sınıfların temel bir soyutlamasını sağlamak için kullanılır. Abstract class'lar, hem imzaları hem de bazı yöntem gövdelerini içerebilir.
Teklik/Multiple Inheritance: Bir sınıf, birden fazla interface'i uygulayabilir (multiple inheritance), ancak bir sınıf yalnızca bir abstract class'ı genişletebilir. Bu nedenle, interface'ler çoklu kalıtım sağlarken, abstract class'lar tek kalıtım sağlar.
Yöntem İçeriği: Interface'lerde, tüm yöntemlerin gövdeleri (implementation) bulunmaz. Yani, bir interface'deki yöntemler sadece imzaları (signature) olarak tanımlanır. Bunun yerine, interface'leri uygulayan sınıflar, bu yöntemlerin gövdelerini kendi içerisinde sağlar. Abstract class'lar ise hem soyut (abstract) yöntemlere hem de somut (concrete) yöntemlere sahip olabilir. Soyut yöntemlerin gövdeleri bulunmaz ve uygulayıcı sınıf tarafından gerçekleştirilmelidir. Somut yöntemler ise abstract class içinde tanımlanmış ve varsayılan bir implementasyonu bulunabilir.
Oluşturma ve Genişleme: Interface'ler, interface anahtar kelimesiyle tanımlanır ve bir sınıf tarafından implements anahtar kelimesiyle uygulanır. Abstract class'lar ise abstract anahtar kelimesiyle tanımlanır ve bir sınıf tarafından extends anahtar kelimesiyle genişletilir.
Kullanım Sıklığı: Interface'ler, daha sık kullanılan bir soyutlama mekanizmasıdır. Özellikle farklı sınıfların aynı davranışı sergilemesini sağlamak ve birlikte çalışabilirlik (interoperability) sağlamak için sıkça kullanılır. Abstract class'lar ise daha özel durumlarda kullanılır ve genellikle daha spesifik bir soyutlama sağlar.
Genel olarak, eğer farklı sınıfların aynı davranışı sergilemesini sağlamak istiyorsanız ve çoklu kalıtım ihtiyacınız varsa, interface'leri tercih etmek daha uygundur. Abstract class'lar ise daha çok ortak davranışı paylaşan sınıfların temel bir soyutlamasını sağlamak için kullanılır ve yöntemlerin bir kısmının gövdelerini içerebilir.

33. String, StringBuilder, StringBuffer ?

String
String her şeyden önce bir sınıftır. En basit tanımıyla char dizilerini temsil eder diyebiliriz. String ilkel bir veri tipi değildir. Bir class’tır ve kendine ait özellikler barındırmaktadır. Konumuz şu an bu olmadığından başka bir yazıda buna değinebiliriz.
Konumuzla ilgili bir String özelliğinden bahsetmek gerekirse, String’ler immutable ’dır. Yani değiştirilemez. Şöyle bir soru sorabilirsiniz: “Ama ben bir String değişken yapıp sonrasında yeni değer ataması yapabiliyorum. O nasıl oluyor?”. Burada yapılan şey aslında var olan String değişkeni değerini değiştirmek olmuyor. Hafızada yeni bir alan açılıyor ve String değişkenimiz artık o alanı göstermiş oluyor. Değişken her güncellendiğinde hafızadan yeni bir alan açılmış oluyor.
bir String değişkenimiz olduğunu düşünün. Belli işlemler yapıyoruz ve her seferinde sonuna + ile elde ettiğimiz diğer verileri ekliyoruz. Yaptığımız bu her + işleminde hafızada yeni bir alan açılmış oluyor. Bu bizim için hem performans kaybı hem de zaman kaybına neden oluyor.
Bu durumda bize yardımcı olacak 2 class mevcut. StringBuilder ve StringBuffer.
StringBuilder Nedir?
StringBuilder sınıfı en kısa tanımla bize “mutable” yani değiştirilebilir string elde etmemize olanak tanır. Böylece hafızada her seferinde yeni bir alan açılmadan var olan alan üzerinde değişiklik yapılabilir. Bu da StringBuilder sınıfını hafıza kullanımı olarak String sınıfının önüne geçirir.
StringBuilder thread-safe değildir. Yani synchronized değildir. Thread’li bir işlem kullanılacaksa StringBuilder kullanılması güvenli değildir. Basit bir şekilde durumu açıklayacak olursak: Aynı anda birden fazla thread, oluşturduğunuz StringBuilder nesnesini değiştirmeye çalıştığında StringBuilder bunu engelleyemez. Bu durumda da threadler arasında yapılan değişiklikler aslında bizim istemediğimiz değer değişikliğine neden olur.
Bir mail göndermek için StringBuilder ile metin oluşturduğunuzu düşünün. Bu işlemi de aynı zamanda 2 farklı thread çalıştırıyor olsun. İlk thread işlemine başlayıp StringBuilder nesnenizi değiştiriyor ve mail içeriğini ayarlıyor. Aynı zamanda diğer thread de aynı StringBuilder nesnesi üzerinde değişiklik yapmak istiyor. Farklı bir kişiye gönderilecek mail için içerik hazırlıyor. Ancak bunu yaparken ilk thread’imizin atadığı değeri ezmiş oluyor. Bunun önüne geçmek için StringBuffer kullanılmalıdır.
StringBuffer Nedir?
StringBuffer ile StringBuilder aynı metodlara sahiptir. Aynı mantıkla ilerler. Aralarındaki tek fark ise StringBuffer thread-safe yani synchronized ‘tır. Bu durum da StringBuffer’ı thread’li işlemlerde kullanılmasını güvenli yapar. Thread’li işlemlerde güvenli olmasının getirdiği bir dezavantaj da mevcuttur. Bu durum StringBuffer’ın StringBuilder’dan daha yavaş çalışmasına neden olur.
Özet
String nesneleri değiştirilemez. Bundan dolayı sürekli üzerinde değişiklik/ekleme yapılacak stringlerimiz varsa hafıza konusunda sıkıntı çıkartması mümkün. Bunun önüne geçmek için StringBuilder ve StringBuffer sınıfları mevcut. Bu iki sınıf değiştirilebilir stringler kullanmamızı sağlıyor. StringBuffer’ın StringBuilder’dan farkı ise çok thread’li ortamlarda çalışırken nesnelerin değişmeyeceği garantisini vermesi. Bu durum da StringBuffer’ın, StringBuilder’dan daha yavaş olmasına neden olmakta.

35. What is Wrapper class ?

Java-da sarğı sinfi primitiv məlumat növünün obyekt yönümlü təqdimatını təmin edən sinifdir. Sarma sinifləri Java API-nin bir hissəsidir və ibtidai məlumat növləri və obyektlər arasında çevrilmə yolunu təmin edir. Wrapper class, Java'da ilkel (primitive) veri tiplerinin nesne olarak temsil edilmesini sağlayan sınıflardır. Java'da her bir ilkel veri tipi için bir wrapper class bulunur. Wrapper class'lar, ilkel veri tiplerini nesne olarak kullanmamızı sağlar ve bu sayede ilkel veri tiplerini objelerin kullanabildiği genel Java API'lerinde kullanabiliriz.
İşte Java'da kullanılan bazı wrapper class'lar ve ilgili ilkel veri tipleri:
Integer: int veri tipinin wrapper class'ıdır.
Double: double veri tipinin wrapper class'ıdır.
Boolean: boolean veri tipinin wrapper class'ıdır.
Character: char veri tipinin wrapper class'ıdır.
Byte: byte veri tipinin wrapper class'ıdır.
Short: short veri tipinin wrapper class'ıdır.
Long: long veri tipinin wrapper class'ıdır.
Float: float veri tipinin wrapper class'ıdır.
Wrapper class'lar, ilkel veri tiplerini nesne olarak temsil ettiği için ilkel veri tipleriyle ilgili bazı ek özelliklere ve yöntemlere sahiptirler. Örneğin, Integer wrapper class'ı, ilkel int değerlerini tutarken aynı zamanda ilkel veri tipleriyle ilgili dönüşüm, karşılaştırma ve matematiksel işlemler yapma gibi yöntemlere de sahiptir.
Wrapper class'lar genellikle aşağıdaki senaryolarda kullanılır:
Bir ilkel veri tipini nesne olarak kullanmak istediğimizde (örneğin, bir koleksiyona eklemek veya bir metodun parametresi olarak kullanmak).
Bir ilkel veri tipinin ekstra özelliklerine veya yöntemlerine ihtiyaç duyduğumuzda.
Bir ilkel veri tipini null değeriyle temsil etmek istediğimizde (null değeri ilkel veri tiplerine atanamaz, ancak wrapper class'ları null değeriyle kullanabiliriz).
Wrapper class'lar otomatik kutulama (autoboxing) ve otomatik açma (unboxing) özelliklerine sahiptir, bu da ilkel veri tipleriyle wrapper class'ları arasında otomatik dönüşümler sağlar. Bu sayede, ilkel ve wrapper veri tipleri arasında kolayca dönüşüm yapabiliriz.

39. What is the difference between an ArrayList and a LinkedList?

In Java, ArrayList and LinkedList are two popular implementations of the List interface, which represents an ordered collection of elements that allows duplicates. Here are the main differences between the two: Java-da ArrayList və LinkedList List interfeysinin iki məşhur tətbiqidir və dublikatlara imkan verən elementlərin sifarişli kolleksiyasını təmsil edir. İkisi arasındakı əsas fərqlər bunlardır:
1. Performance characteristics: ArrayList is faster than LinkedList for random access of elements, as it can directly access any element using its index. HashMap, açarla elementlərə sürətli daimi daxil olmağa imkan verən hash cədvəlindən istifadə etməklə həyata keçirilir. HashMap-dakı elementlərin sırasına zəmanət verilmir və elementlər əlavə olunduqca və ya silindikcə dəyişə bilər.
TreeMap isə qırmızı-qara ağac kimi həyata keçirilir ki, bu da elementləri açarlarına görə sıralamağa imkan verir. TreeMap-dakı elementlərin sırası onların təbii sırasına və ya təqdim olunarsa, fərdi müqayisəçiyə əsaslanır.
The choice between HashMap and TreeMap depends on the specific requirements of the use case. If fast access to elements by key is important and order is not relevant, HashMap is a good choice. If ordering by key is important, TreeMap is a better choice.
In terms of performance, HashMap generally offers faster constant-time access for operations such as get and put, but may be slower than TreeMap for operations such as iteration or ordered traversal. TreeMap, on the other hand, has a slower constant time access but offers faster ordered traversal and iteration.
Performans baxımından, HashMap ümumiyyətlə almaq və qoymaq kimi əməliyyatlar üçün daha sürətli daimi giriş təklif edir, lakin iterasiya və ya sifarişli keçid kimi əməliyyatlar üçün TreeMap-dən daha yavaş ola bilər. Digər tərəfdən TreeMap, daha yavaş daimi vaxt girişinə malikdir, lakin daha sürətli sifarişli keçid və iterasiya təklif edir.

45. What is Dependency Injection ?

Dependency Injection (DI) Sınıflar birbirlerini kullanarak yapılacak olan işlemleri gerçekleştirirler. Bu sınıflar arası bağımlılıkların oluşmasını sağlar.
public class UserController { private UserServiceImpl userServiceImpl = new UserServiceImpl(); } Yukarıdaki örnekte UserController sınıfı belirli işlemleri gerçekleştirebilmek için UserServiceImpl sınıfına bağımlıdır. Fakat UserController sınıfı içerisinde new operatörü ile nesne oluşturulması, UserController sınıfını direkt olarak UserServiceImpl sınıfına bağımlı kılmaktadır. Bu Dependency Inversion Principle’a aykırıdır. Çünkü bağımlılığın yönünün soyut sınıflara doğru olması gerekirken burada somut bir sınıfa doğru olmuştur. Herhangi bir şekilde UserServiceImpl sınıfı değişikliğe uğrarsa, UserController bu değişiklikten direkt olarak etkilenecektir.
(Yuxarıdakı nümunədə UserController sinfi müəyyən əməliyyatları yerinə yetirmək üçün UserServiceImpl sinfindən asılıdır. Bununla belə, UserController sinfində new operatorla obyekt yaratmaq UserController sinfini birbaşa UserServiceImpl sinfindən asılı edir. Bu, Asılılığın İnversiya Prinsipinə ziddir. Çünki asılılığın istiqaməti mücərrəd siniflərə doğru olmalıdır, amma burada konkret sinfə doğrudur. UserServiceImpl sinfi hər hansı şəkildə dəyişərsə, UserController bu dəyişiklikdən birbaşa təsirlənəcək.)
public class UserController { private UserService userService; public UserController(UserService userService) { this.userService = userService; } } Yukarıdaki örnekte ise UserController sınıfı bu sefer somut bir sınıfa bağımlı olmamakla birlikte new operatörünü kullanarak bağımlılığını kendisi oluşturmuyor. Peki bu bağımlılık nasıl oluşuyor? UserController sınıfı oluşturulduğu anda constructor metoduna uygulama çatısı tarafından herhangi bir UserService implementasyonu gönderilerek bu bağımlılık enjekte ediliyor. Bu işleme Dependency Injection yani bağımlılıların enjekte edilmesi denir.

48. What is Ant, Maven, Gradle

Maven : Maven nedir? Maven özetle Apache tarafından geliştirilmiş, yazılım projelerinizin geliştirilmesi sırasında projenin geliştirilmesini kolaylaştıran, basitleştiren ve proje bağımlılıklarının kontrolünü yapmayı sağlayan bir araçtır(pek özet olmadı!). Maven projedeki bağımlılıkların(dependency) yönetimi dışında IDE(Integrated Development Environment) bağımsız yazılım geliştirmeyi de amaçlar.
Ant, Maven, and Gradle are build automation tools used in Java development. They help manage the build process, including compiling source code, resolving dependencies, and packaging the application into a distributable format. Here's a brief explanation of each:
Ant, Maven və Gradle Java inkişafında istifadə olunan build avtomatlaşdırma tooldur. Onlar mənbə kodunun tərtib edilməsi, asılılıqların həlli və tətbiqin paylana bilən formatda qablaşdırılması daxil olmaqla, tikinti prosesini idarə etməyə kömək edir. Budur hər birinin qısa izahı:
Ant: Ant (Apache Ant) is an older build tool that uses XML-based configuration files to define the build process. It provides a flexible and extensible way to automate the build, test, and deployment tasks. Ant allows developers to define targets and tasks for compiling, testing, packaging, and other build-related activities. It does not have built-in dependency management and relies on external libraries or manual configuration for that purpose.
Maven: Maven is a popular build automation tool that focuses on standardizing the project structure, dependency management, and build lifecycle. It uses an XML-based Project Object Model (POM) file to define the project configuration. Maven provides a set of predefined lifecycle phases (e.g., compile, test, package) and plugins to perform various tasks. It manages dependencies automatically by resolving them from remote repositories. Maven promotes convention over configuration, which means it follows predefined conventions to simplify project setup and reduce configuration.
Gradle: Gradle is a modern build automation tool that combines the best features of Ant and Maven while introducing a more flexible and expressive Groovy-based DSL (Domain-Specific Language). Gradle builds are written in Groovy or Kotlin, allowing developers to define custom build logic using a concise and readable syntax. It supports incremental builds, parallel execution, and dependency management through its powerful dependency resolution engine. Gradle is highly customizable and scalable, making it suitable for small to large projects.
Key differences:
Configuration: Ant uses XML configuration, Maven uses XML-based POM files, and Gradle uses Groovy or Kotlin DSL. Gradle's DSL provides more flexibility and expressiveness compared to XML.
Yapılandırma: Ant, XML yapılandırmasını kullanır, Maven, XML tabanlı POM dosyalarını kullanır ve Gradle, Groovy veya Kotlin DSL kullanır. Gradle'ın DSL'si, XML'e kıyasla daha fazla esneklik ve ifade sağlar.
Dependency Management: Ant does not have built-in dependency management, while both Maven and Gradle handle dependency resolution from remote repositories. Maven has a centralized repository and enforces a strict dependency structure, while Gradle offers more flexibility in managing dependencies.
Asılılığın İdarə Edilməsi: Ant daxili asılılıq idarəetməsinə malik deyil, həm Maven, həm də Gradle uzaq depolardan asılılıq həllini idarə edir. Maven mərkəzləşdirilmiş depoya malikdir və ciddi asılılıq strukturunu tətbiq edir, Gradle isə asılılıqların idarə edilməsində daha çox çeviklik təklif edir.
Convention vs. Configuration: Maven follows convention over configuration principles, providing a predefined project structure and build lifecycle. Gradle allows more customization and lets developers define their own project structure and build logic.
Learning Curve: Ant has a steeper learning curve due to its XML configuration. Maven provides a simpler and standardized approach, while Gradle offers a more powerful and flexible but slightly more complex DSL.
Öyrənmə əyrisi: Ant XML konfiqurasiyasına görə daha dik öyrənmə əyrisinə malikdir. Maven daha sadə və standartlaşdırılmış yanaşma təqdim edir, Gradle isə daha güclü və çevik, lakin bir qədər daha mürəkkəb DSL təklif edir.
Ultimately, the choice between Ant, Maven, or Gradle depends on the project requirements, team preferences, and the ecosystem you are working with. Maven and Gradle are more commonly used and have extensive plugin support, making them popular choices for Java projects.

CALL — PL/SQL veya Java alt programı çalıştırır.

DCL

Data Control Language izin ve yetkilendirme işleri için kullanılır.
GRANT — Kullanıcıya veritabanı erişim yetkisi verir.
REVOKE — GRANT ile verilen yetkiyi geri alır.

TCL

Transaction Control DML ile yapılan işlemlerin yönetilmesini sağlar.
COMMIT — Yapılanları kayıt eder.
SAVEPOINT — Daha sonra geri dönülecek bir dönüş noktası belirler.
ROLLBACK — Son COMMIT’e kadar olan yeri geri alır.
